# Sprint 1 — Criteria Alignment: Theme engine + no-FOUC + site-wide token migration

Status: **aligned**
UTC: 2026-06-27 03:43:36
Scope: theme engine only — no settings UI. Theme is exercised via `data-theme`
on `<html>` and/or seeding `localStorage.theme` then reloading. All criteria are
verified by interacting with the running site (computed styles / DOM /
screenshots / frame diffs) at **1280x900** and **390x844**, across **all three
themes** (light, dark, sunset) on every page in scope.

Pages in scope: `/` (home), `/about`, `/posts`, a post detail
`/posts/[slug]`, `/travel` (incl. the WebGL globe + its label/legend). Surfaces
in scope: sticky header, hamburger menu (SiteMenu), mobile tab bar.

---

## Agreed acceptance criteria (6)

1. **Theme switching re-colors the entire site, and the migration is complete.**
   Setting `<html data-theme>` to `light` / `dark` / `sunset` (or seeding
   `localStorage.theme` then reloading) re-colors **every** page and chrome
   surface in scope. Verifiable per theme: `getComputedStyle(body).backgroundColor`
   matches the theme's `--bg`, and the header, hamburger menu, and mobile tab bar
   backgrounds + text track the theme too. **Completeness gate:** in dark and in
   sunset, no in-scope page or surface still renders a hardcoded light value —
   specifically there is no element with computed `background-color` equal to
   `rgb(255,255,255)`/light-gray (`#f3f4f6`) or text/border equal to a light-mode
   literal (`#1a1a1a` fg, `#6b7280` muted, `#e5e7eb` border, `#2563eb` blue
   accent) that does not belong to that theme's palette. Spot-checked on home,
   about, posts, a post page, travel, header, menu, and tab bar.

2. **Genuinely no flash of the wrong theme (FOUC), on reload and on navigation.**
   With `localStorage.theme="dark"` (or `"sunset"`) set before load, the **first
   paintable frame already shows that theme** — `document.documentElement` has the
   correct `data-theme` when read immediately on load (the blocking inline head
   script ran before paint), and a screenshot taken at first paint shows the
   themed background, never a white flash. The same holds when navigating between
   two in-scope pages (e.g. `/` → `/travel`) with a stored non-default theme: no
   intermediate white/light frame.

3. **OS default before choice; explicit choice persists across reload AND
   navigation.** With no stored `theme` and the emulated OS scheme = dark, the
   site renders dark; with emulated OS scheme = light, it renders light (the
   default follows `prefers-color-scheme`). After an explicit theme is applied
   (via the provider API / `localStorage.theme`), that theme survives a full
   reload and survives navigating from one in-scope page to another and back —
   `data-theme` and the rendered colors stay on the chosen theme throughout.

4. **Body and muted text are measurably legible in all three themes, every page,
   both viewports.** On home, about, posts, a post page, and travel, primary body
   text contrast vs its background is comfortable (target WCAG AA, **≥ 4.5:1**),
   and secondary/muted text (dates, captions, the globe label hint) stays clearly
   readable (**≥ 3:1**, never near-invisible). Accent-colored links are
   distinguishable from body text and meet **≥ 3:1** vs their background in each
   theme. Verified by computing contrast from `getComputedStyle` on representative
   text nodes per page/theme, corroborated by screenshot review. No theme has
   near-invisible text on any in-scope page.

5. **The globe is theme-aware and legible in all three themes — not just light.**
   On `/travel`, the WebGL globe's sphere, country borders, and pins are derived
   from the active theme so the globe reads clearly in dark and sunset, not only
   light: the sphere is **not** a bright pale-blue (`#cfe0f5`) patch floating on a
   dark/warm background, borders and resting pins are visible against the sphere,
   and the active pin stays a distinct warm highlight clearly separable from
   resting pins. Verified by visible difference in the globe canvas region between
   the three themes (frame diff of the canvas across themes is non-trivial), pins
   observable, and the label/legend text under the globe migrated to themed
   `text-fg` / `text-muted` (legible per criterion 4). Switching theme while on
   `/travel` recolors the globe without a reload.

6. **No regressions; clean and green.** Globe still auto-rotates with no input and
   is draggable (a drag visibly rotates it), pins still hover/select and reveal a
   country name + date, the sticky header still sticks on scroll, and there is **no
   horizontal overflow at 390px** (`document.documentElement.scrollWidth <=
   window.innerWidth`) on any in-scope page in any theme. **Zero** console errors
   / page errors on every in-scope page (all three themes). All rendered `img`
   have `naturalWidth > 0`. `pnpm build` exits 0 (static export succeeds).

## Rationale — one line per change from the DRAFT

- **Draft 1 → Agreed 1:** Kept the "re-colors whole site" outcome but added a
  falsifiable **migration-completeness gate** (no surviving light-mode literal
  colors in dark/sunset) — the draft's "colors differ across themes" can pass
  even if a page silently keeps `text-gray-*`/`bg-white`, which is the sprint's
  primary failure mode.
- **Draft 2 → Agreed 2:** Strengthened no-FOUC to require the **themed background
  on the first paint frame** (not merely the attribute set) and extended it to
  **navigation between pages**, not just reload, per GOAL "no flash on load" +
  "across navigations."
- **Draft 3 → Agreed 3:** Kept OS-default + persistence; made persistence
  explicitly cover **both reload and navigation** and both OS directions
  (dark→dark, light→light) so the default truly follows `prefers-color-scheme`.
- **Draft 4 (split) → Agreed 4:** Turned "clearly readable / comfortable
  contrast" into **measurable thresholds** (AA ≥ 4.5:1 body, ≥ 3:1 muted/accent)
  per SPEC §3, applied to **every** in-scope page at **both** viewports — removing
  the "screenshot vibes only" loophole while keeping screenshot corroboration.
- **Draft 4 (split) → Agreed 5:** Promoted the globe to its **own criterion** with
  observable assertions (sphere not pale-blue-on-dark, borders/pins visible,
  active pin distinct, label migrated, recolors on theme change). The globe is the
  highest-risk surface (literal hex baked into WebGL) and GOAL calls it out
  explicitly; burying it in a shared legibility line under-weights it.
- **Draft 5 → Agreed 6:** Kept as the regression/green-gate criterion; made the
  no-overflow and zero-console-errors checks apply **per theme** (a theme switch
  can introduce a new error or overflow), and made globe interactivity assertions
  concrete (drag rotates, pin reveals name + date).
- **Redundancy cut:** Folded the draft's scattered build/overflow/console/globe
  checks into a single regression criterion (6) so the list stays at 6 and each
  criterion owns a distinct outcome.
